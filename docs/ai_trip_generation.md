# Smart Trip Generation: Architecture & Data Flow

This document outlines the technical implementation of the Trippzi Smart Trip Generator and its supporting services. The system is designed as a **self-enriching data engine** (Growth Loop) that minimizes API costs while maximizing the realism and accuracy of travel recommendations.

---

## 🏗️ 1. High-Level Architecture & Service Usage

The platform's logic is distributed across four core services. Here is **when** and **how** they are triggered:

### A. `AIEngine` (The Orchestrator)
- **When**: Triggered during the `GenerateItineraryView` (POST) and background tasks.
- **How**: It acts as the central brain. It receives user preferences, gathers context from `SearchService` and `VisaService`, and constructs the final day-by-day JSON structure. It also handles the "Growth Loop" by calling `refresh_country_data` and `refresh_attraction_data`.

### B. `SearchService` (The Real-Time Eye)
- **When**: Triggered at the very start of the generation flow and during metadata refreshes.
- **How**: It uses the **Tavily API** to provide "Web Grounding." It fetches real-time data on ticket prices, trending restaurants, and transit options. This prevents the AI from hallucinating outdated information.

### C. `VisaService` (The Rules Engine)
- **When**: Triggered during `AIEngine.generate_itinerary()`.
- **How**: It checks the `VisaRequirement` matrix. If a rule for `Source -> Destination` is missing or older than **60 days**, it performs a targeted search to update the database. This ensures travelers always see the latest visa requirements for their specific passport.

### D. `ImageService` (The Visual Layer)
- **When**: Triggered during `GenerateItineraryView` to fetch hero images and day-wise activity photos.
- **How**: It uses the **Unsplash API** to find high-quality, landscape imagery based on queries generated by the `AIEngine`. It also handles the secure downloading and saving of these images to your storage (R2/S3).

### E. `PDFService` (The Document Layer)
- **When**: Triggered when a user requests a download for a purchased itinerary.
- **How**: Implements a "Triple Fallback" rendering chain (`WeasyPrint` → `xhtml2pdf` → `fpdf2`). It handles complex HTML-to-PDF conversions, integrates Unsplash images, and manages the final storage in R2/S3. It also implements automatic reuse of existing PDFs to save server resources.

### F. `PaymentService` (The Revenue Layer)
- **When**: Triggered when a user clicks "Buy Itinerary" or completes a payment.
- **How**: 
    - **Order Creation**: Initializes a Razorpay order ID and calculates the amount based on `SiteSettings`.
    - **Verification**: Validates the Razorpay signature after the user pays to ensure the transaction is authentic before marking the itinerary as "purchased."

## 🔄 2. The Generation Lifecycle (Sequence)

1.  **User Input**: User submits a country, budget, and travel style.
2.  **Data Warming**: `AIEngine` checks if Country/Destination data exists. If not, it calls `SearchService`.
3.  **Context Assembly**: `AIEngine` calls `SearchService` for city-specific trends and `VisaService` for passport-specific rules.
4.  **AI Synthesis**: The prompt is sent to OpenAI with all gathered context.
5.  **Post-Processing**: `AIEngine` saves new `Attraction` objects and triggers the 60-day refresh if they already existed.
6.  **Monetization**: Once the itinerary is ready, the `PaymentService` handles the transition from "Draft" to "Purchased."

## 💾 3. Data Persistence & Caching Rules

| Model | Service Responsible | Caching Duration |
| :--- | :--- | :--- |
| **Country Metadata** | `AIEngine` + `SearchService` | Permanent (until forced) |
| **Visa Matrix** | `VisaService` | **60 Days** |
| **Attraction Details**| `AIEngine` + `SearchService` | **60 Days** |
| **Transactions** | `PaymentService` | Permanent (Audit Log) |

## 🚄 4. Realistic Transfers & Logistics

The `AIEngine` uses a specific instruction set to ensure transit is realistic:
- **Tavily Grounding**: The engine recommends specific modes (e.g., "Take the BTS Skytrain," "Walking," or "15min Taxi") based on the city's actual infrastructure fetched by `SearchService`.
- **Transfer Logic**: It calculates `distance_to_next` and `time_to_next` based on real-world coordinates and traffic patterns provided in the search context.

---

## 🖥️ 5. UI Integration

- **User Portal**: Consumes `AIEngine` output for the itinerary view and `PaymentService` for the checkout modal.
- **Superadmin**: Uses the `AIEngine` "Growth Loop" data to display a growing database of global travel knowledge.
- **Visa Tab**: Directly displays the content managed by `VisaService`.
